# 执行摘要
## 优化目标
据用户反馈，Atypica当前的报告不够有“深度”。经过进一步追问，我们对“有深度的报告”进行了拆解：
1. 信息来源可靠且可追溯
2. 报告内容从上到下有清晰的逻辑链路/方法论，而不是信息的堆砌
3. 读者认同报告所采用的逻辑链路/方法论
4. 报告的最终建议/产出足够具体且有实操性

# 阶段性优化过程
## 1. 定制商业化分析框架
## 问题定义
- 当前报告内容只是用户访谈内容的堆砌，没有用户能够理解和认同的清晰逻辑链路/方法论
## 优化方案
- 使用如Job-to-be-done、KANO等知名商业化框架，定制成独立的报告生成提示词。
- Atypica会基于用户的问题来判断调用什么商业化框架来完成报告
- 先挑选了一个商业化框架：Job-to-be-done（JTBD）来做测试

## 2. （标题没想好）
## 问题定义
- 有框架，但是内容不够具体。这是因为报告生成过程中拿到的输入只有interview的过程和研究问题的概括，没有能够补充到商业化分析框架里的“具体内容”————也就是基于当前分析框架的研究分析过程。
- 报告的框架感太强，感觉在强行套用框架。这是因为商业化分析框架适用于研究分析过程，而不是报告的呈现方式。（eg. JTBD是个适合用来指导分析的框架，而不适合用来指导最终报告）所以研究分析过程应该根据用户问题选择最适合的商业化分析框架，而报告的呈现方式应该从用户视角优化用户阅读的逻辑链路。
## 优化方案
1. 为了解决商业分析框架和用户问题的适配，添加了一位商业化分析专家智能体。它的任务是根据用户的问题，推荐最适合用来分析这个问题的商业化分析框架，并指导Atypica进行研究的信息收集和信息分析。详细指导内容包括：
    - 方法论推荐：想要得到这个结果，应该使用什么专业的商业分析框架？为什么适合？<节选>
    - 执行建议-信息收集：基于这个分析框架，应该如何通过了联网搜索和用户访谈来获得分析所需要的信息？<节选>
    - 执行建议-信息分析：基于这个分析框架，应该如何对收集的信息进行专业的分析？<节选>
2. 为了解决缺少研究分析过程的问题，给Atypica添加了“深度分析”的能力。Atypica现在在报告生成前，会将收集到的联网搜索结果和用户访谈结果一并整理，并基于商业化分析专家的建议进行符合商业化分析框架的信息推理和信息分析。整个分析的复杂过程和结果将会用于报告生成。

## 3. 更有实操性和同理心的报告产出
## 问题定义
- 整个研究报告中的信息有三个：研究问题，研究过程，研究结果/产出。对用户来说最关心的是研究结果/产出>研究过程>研究问题。但是现在的报告呈现的重点是研究过程>研究问题>研究结果/产出，和用户的期待有gap，导致了用户看完报告觉得“这个报告没什么用”的结果。
## 优化方案
- 整个研究分析过程应该是面向着用户期待中的产出进行的，这样推荐出来的分析框架才是用户心目中认同的。商业化分析专家智能体会先深刻理解用户的画像和他最终想要的产出具体是什么？eg 产品方案、投放策略、定价策略
- “深度分析“中要求最终产出要更具体，并且遵循商业化分析专家智能体对产出形式的建议。
- 报告生成的提示词进行了修改：报告的撰写框架应该是一个倒叙逻辑：先简单的汇报研究问题，研究过程，研究结果/产出（重点）。

- Token消耗因为添加了两个很重的能力（商业化分析专家智能体、“深度分析“能力）变得更多了，所以token的消耗有很大的优化空间

- 报告中缺少联网搜索的来源追溯
