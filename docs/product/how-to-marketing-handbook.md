## 1. 整体方法论

### 1.1 为什么需要这套方法？

**痛点**：
- 技术团队写的文档太技术，用户看不懂
- 营销团队写的内容太虚，没有具体证据
- 功能更新后，营销材料没及时同步，出现错误信息

**解决方案**：
建立"代码 → 功能 → 传播"的完整链条，确保：
1. **唯一真相来源**：所有营销材料基于同一份真相文档
2. **可验证性**：每个陈述都有代码/数据证据
3. **及时同步**：功能更新时，只需更新真相文档，传播材料自动同步

### 1.2 三层架构

```
┌─────────────────────────────────────────┐
│  层次3：传播内容（夺人眼球）              │
│  官网落地页、KOL演讲稿、社交媒体内容       │
└─────────────────┬───────────────────────┘
                  │ 基于
┌─────────────────▼───────────────────────┐
│  层次2：真相文档（客观、细节、不啰嗦）     │
│  功能对比、竞品分析、能力边界清单          │
└─────────────────┬───────────────────────┘
                  │ 提炼自
┌─────────────────▼───────────────────────┐
│  层次1：代码和产品（已实现的功能）         │
│  Agent配置、工具实现、系统提示词          │
└─────────────────────────────────────────┘
```

**关键原则**：
- 层次1是"源头"，不能凭空想象
- 层次2是"翻译"，把技术语言翻译成人话
- 层次3是"包装"，让人话变得吸引人

### 2.2 提炼关键信息（回答8个问题）

为每个功能回答以下问题：

#### Q1: 这个功能解决什么问题？
**方法**：从系统提示词找"适用场景"
**示例**：
- Interview：追踪个体完整决策旅程和使用流程
- Discussion：观察用户在多个方案间的权衡辩论

#### Q2: 核心差异是什么？
**方法**：列出所有可对比维度
**示例**：

| 维度 | Interview | Discussion |
|------|-----------|-----------|
| 交互模式 | 一对一串行 | 多人并行 |
| 人设数量 | 5-10个 | 3-8个 |
| 工具返回 | 不返回 | 返回总结 |
| 主Agent可见 | 完全不可见 | 可见总结 |

#### Q3: 技术参数是多少？
**方法**：从代码找硬编码的数字
**示例**：
```typescript
// Interview
personaIds: string[],  // 5-10个
批次限制：每次最多 5 人

// Discussion
personaIds: string[],  // 3-8个
无批次限制
```

#### Q4: 有什么限制？
**方法**：找代码中的 validation、error handling
**示例**：
- Interview：禁止重复访谈同一人设（系统检测并跳过）
- Discussion：必须提供详细 instruction 参数

#### Q5: 为什么这样设计？
**方法**：读架构文档、代码注释
**示例**：
- Interview 不返回结果：因为访谈内容太长，直接进报告系统更高效
- Discussion 返回总结：因为总结可用于后续决策

#### Q6: 和什么功能容易混淆？
**方法**：找功能名称相似或场景重叠的
**示例**：
- Interview vs Discussion（都是用户研究）
- Scout vs Study Agent（都可以做观察）
- Fast Insight vs 常规研究（都能生成报告）

#### Q7: 典型使用场景是什么？
**方法**：从用户旅程文档找真实案例
**示例**：
- Interview：气泡咖啡案例中的深度访谈
- Discussion：焦点小组讨论"方案A vs 方案B"

#### Q8: 我们能做什么、不能做什么？
**方法**：区分技术限制和战略选择
**示例**：

**能做**：
- ✅ 模拟5-10个AI人设进行深度访谈
- ✅ 自动追问"为什么"（无限深度）

**不能做（技术限制）**：
- ❌ 不能访谈真人（只能模拟AI人设）
- ❌ 不能显示访谈过程给用户（设计选择）

**不能做（战略选择）**：
- ❌ 不做视频访谈（定位是文本研究）
- ❌ 不做实时协作（避免功能膨胀）

```markdown
# Interview vs Discussion：什么时候用哪个？

## 一句话差异
Interview是一对一深度挖掘个体动机，Discussion是群体观点碰撞找共识

## 核心差异表格

| 维度 | Interview（一对一访谈） | Discussion（群体讨论） |
|------|------------------------|----------------------|
| **适用场景** | 追踪个体完整决策旅程、挖掘深层动机 | 观察群体权衡过程、意见分布 |
| **人数** | 5-10个AI人设 | 3-8个AI人设 |
| **批次限制** | 每批最多5人（超过需分批） | 无批次限制，一次性讨论 |
| **交互模式** | 一对一串行访谈 | 多人并行讨论 |
| **输出形式** | **不返回访谈结果** | **返回讨论总结** |
| **用户可见性** | 完全不可见，只有最终报告 | 可见总结，不可见完整对话 |
| **时长** | 约15-20分钟（取决于人设数量） | 约10-15分钟 |
| **核心价值** | 深度个体洞察 | 群体动态观察 |

## 详细对比

### Interview的特点
- **设计目的**：深入了解个体的完整经历、情感、决策过程
- **工作原理**：AI访谈员与每个人设一对一对话，深度追问"为什么"
- **关键参数**：
  - `personaIds`：5-10个（标准），可根据研究需要调整
  - `instruction`：访谈指令，指导访谈员问什么
  - 批次限制：每次最多5人，超过需分批（避免超时）
- **输出特征**：
  - 工具**不返回任何内容**
  - 访谈数据直接进入系统，用于报告生成
  - 主Agent看不到访谈过程

### Discussion的特点
- **设计目的**：观察群体如何讨论、辩论、达成共识
- **工作原理**：多个人设同时在虚拟会议室讨论，AI主持人引导
- **关键参数**：
  - `personaIds`：3-8个（需要对比性观点）
  - `instruction`：必须详细（包含讨论目的、背景信息、期望风格）
  - 无批次限制
- **输出特征**：
  - 工具**返回讨论总结**
  - 完整讨论内容系统记录，用于报告
  - 主Agent可看到总结，可基于总结继续决策

## 使用建议

### 什么时候用Interview？

✅ **场景1：探索"为什么"类问题**
   - 示例："用户为什么放弃购物车？"
   - 原因：需要深入挖掘情感、动机、心理障碍

✅ **场景2：追踪完整决策旅程**
   - 示例："从了解产品到最终购买，完整过程是怎样的？"
   - 原因：需要了解每个阶段的想法变化

✅ **场景3：涉及隐私敏感话题**
   - 示例："个人理财习惯和焦虑"
   - 原因：一对一更容易获得真实反馈

✅ **场景4：需要详细的个人故事**
   - 示例："描述你最近一次使用我们产品的完整体验"
   - 原因：群体讨论容易跑题，个体访谈更聚焦

### 什么时候用Discussion？

✅ **场景1：比较多个方案**
   - 示例："方案A vs 方案B，哪个更好？"
   - 原因：群体讨论能看到不同观点的碰撞

✅ **场景2：模拟焦点小组**
   - 示例："新产品概念测试会"
   - 原因：需要观察群体反应和互相影响

✅ **场景3：寻找共识点和分歧点**
   - 示例："不同年龄段用户对价格的接受度"
   - 原因：讨论中自然显示意见分布

✅ **场景4：观察用户互相说服的过程**
   - 示例："为什么最终大家选择了方案A？"
   - 原因：群体动态比个体陈述更真实

### 不确定？问自己

- [ ] 我关心的是**个体的深层原因**吗？ → Interview
- [ ] 我关心的是**群体的意见分布**吗？ → Discussion
- [ ] 问题涉及**私密或敏感话题**吗？ → Interview
- [ ] 需要看**观点碰撞和说服过程**吗？ → Discussion
- [ ] 我要追踪**一个人的完整旅程**吗？ → Interview
- [ ] 我要看**不同类型用户的权衡**吗？ → Discussion

## 技术实现差异

**Interview**：
```typescript
interviewChatTool({
  personaIds: string[],     // 5-10个
  instruction: string       // 访谈指令
})
// 返回值：无（undefined）
// 访谈内容直接写入数据库，用于报告生成
```

**Discussion**：
```typescript
discussionChatTool({
  personaIds: string[],     // 3-8个
  instruction: string       // 必须详细
})
// 返回值：{ summary: string }
// 完整讨论内容写入数据库，summary可被主Agent使用
```

## 常见误解

❌ **误解1**："Interview更准确，Discussion不靠谱"
✅ **真相**：两者目的不同。Interview适合深度挖掘，Discussion适合观察群体动态，没有谁更准确

❌ **误解2**："用Discussion更快，因为一次性完成"
✅ **真相**：时间差不多。Interview虽然串行，但每个访谈更深入；Discussion虽然并行，但需要AI主持协调

❌ **误解3**："Interview可以看到访谈过程"
✅ **真相**：Interview不返回任何内容，访谈过程对主Agent和用户都不可见（设计选择，确保数据一致性）

❌ **误解4**："Discussion的讨论内容可以直接看到"
✅ **真相**：只能看到总结（summary），完整讨论内容系统记录但不直接展示

## 组合使用

**推荐组合1：Scout → Interview**
1. 先用Scout观察社交媒体，找到目标人群
2. 再用Interview深度访谈，挖掘深层动机

**推荐组合2：Interview → Discussion**
1. 先用Interview了解个体观点
2. 再用Discussion看群体如何达成共识
（注意：目前系统不支持这个流程，一个研究只能选一种方式）

**不推荐**：同一个研究中既用Interview又用Discussion
- 原因：两者数据结构不同，报告生成逻辑会混乱
- 替代方案：做两次独立研究

## 5. 质量标准与检查

### 5.1 客观性标准

**黄金标准**：所有陈述可验证

#### 检查方法：
1. 每个数据都能找到来源（代码/文档/数据）
2. 竞品对比中承认了对方优势
3. "能做"和"不能做"都清楚说明
4. 没有"最好"、"唯一"、"完美"等表述
5. 每个优势都有具体证据

#### 常见错误：
❌ "atypica是最好的研究工具"
✅ "atypica在速度上有优势（30分钟 vs 2-4周）"

❌ "我们的AI人设完全等同于真人"
✅ "深访人设一致性85分，超越人类基线81%"

❌ "竞品X功能很差"
✅ "竞品X在文档理解上更专业，atypica聚焦主观研究"

### 5.2 可读性标准

**黄金标准**：3秒抓住重点

#### 检查方法：
1. 核心信息在表格中（不是段落）
2. 使用具体场景和例子（不是抽象描述）
3. 技术细节简洁（≤200字）
4. 避免行话（用人话）
5. 段落短（≤3-4句）

#### 常见错误：
❌ 长段落堆砌信息
✅ 用表格 + bullet points

❌ "atypica采用messages as source of truth架构..."
✅ "atypica会记住你的公司背景，下次无需重复说明"

❌ "通过向量相似度搜索匹配人设..."
✅ "从30万人设库中自动找到最相关的用户"

### 5.3 实用性标准

**黄金标准**：用户能立即行动

#### 检查方法：
1. 用户能快速判断"我该用哪个"
2. 有明确的使用建议
3. 有"不确定时"的判断清单
4. 有组合使用建议（如适用）
5. 有常见误解澄清

#### 常见错误：
❌ 只说功能，不说场景
✅ "如果你要比较方案A和B → 用Discussion"

❌ 让用户自己判断
✅ "不确定？问自己：我关心个体深层原因吗？→ Interview"

## 6. 常见问题解答

### Q1：如何验证竞品信息的准确性？

**方法1：实际试用**（最准确）
- 注册竞品账号，实际体验功能
- 记录关键参数（价格、功能、限制）
- 截图保存证据

**方法2：官网和文档**（快速但可能滞后）
- 访问官网查看最新信息
- 阅读产品文档和定价页
- 注意：可能不是最新，需要标注时间

**方法3：用户评价**（真实但主观）
- G2、Capterra等评价网站
- Reddit、HN等社区讨论
- 注意：主观性强，需要多方验证

**方法4：询问用户**（如果有）
- 询问实际用过的用户
- 问具体问题（不是"好不好用"）
- 注意：样本量小，可能有偏差

**最佳实践**：
- P0核心竞品：方法1（实际试用）
- P1相关竞品：方法2+方法3
- P2边缘竞品：方法2

### Q2：功能还在开发中，如何写文档？

**原则**：只写已实现的，明确标注未实现的

**方法**：
1. 在文档中明确区分：
   ```markdown
   ## 当前功能
   ✅ 已实现：Interview (一对一访谈)
   ✅ 已实现：Discussion (群体讨论)

   ## 规划中功能
   🚧 规划中：Video Interview (视频访谈)
   🚧 规划中：Real-time Collaboration (实时协作)
   ```

2. 不要过度承诺：
   ❌ "即将支持视频访谈"（模糊时间）
   ✅ "目前聚焦文本访谈，视频访谈在产品路线图中"

3. 更新及时：
   - 功能上线后立即更新文档
   - 在文档中标注更新日期
   - 维护更新记录

### Q3：用户数据不足，如何写数据支撑？

**方法1：用相对数据代替绝对数据**
❌ "已有10000个用户"（如果实际只有100，不诚实）
✅ "深访人设一致性85分，超越人类基线81%"（相对数据，更有说服力）

**方法2：用技术参数代替运营数据**
❌ "完成10000次研究"（如果实际很少）
✅ "每次研究平均30分钟，使用200万tokens"（技术参数，可验证）

**方法3：用案例代替数量**
❌ "被大量企业使用"（模糊）
✅ "某咖啡品牌用atypica完成气泡咖啡新品验证"（具体案例）

**方法4：坦诚说明**
✅ "atypica处于早期阶段，用户数持续增长中"
✅ "欢迎成为早期用户，获得优先支持"

**原则**：宁可不写，不要编造

### Q4：技术文档太难懂，如何翻译成人话？

**方法：使用类比**

技术术语 → 人话 → 类比

| 技术术语 | 人话 | 类比 |
|---------|------|------|
| Messages as Source of Truth | 消息即真相来源 | 像银行流水账，每笔交易都记录，不会丢 |
| Vector similarity search | 向量相似度搜索 | 像"找相似的人"，系统自动匹配 |
| Streaming response | 流式响应 | 像打字效果，边生成边显示 |
| Tool invocation | 工具调用 | AI会"使用工具"完成任务 |
| Prompt engineering | 提示词工程 | 像给AI写"作业要求" |

**方法：分层说明**

```markdown
## Interview（一对一访谈）

**人话**：AI访谈员和每个人设一对一聊天，深度追问"为什么"

**类比**：像心理咨询师做咨询，一对一深入了解对方

**技术细节**（可选）：
- 使用 `interviewChatTool` 工具
- 工具不返回结果（undefined）
- 访谈内容直接写入数据库
- 系统自动整合到报告中
```

### Q5：不同版本落地页内容差异太大，如何保持一致性？

**方法：建立内容库**

**核心模块（所有版本都有）**：
1. 震撼开场（行业转折点）
2. 核心数据（30万人设、85分一致性）
3. CTA（行动召唤）

**可变模块（根据人群调整）**：
1. 痛点描述（不同人群痛点不同）
2. 使用场景（不同人群场景不同）
3. 案例选择（选择相关案例）

**示例**：

```markdown
# 内容库

## 核心模块（固定）
- 开场：Gartner暴跌30%
- 数据：30万人设、85分一致性、30分钟、50-100元
- CTA：开始你的第一个研究

## 痛点库（按人群）
- Consultants：客户要洞察不要PPT
- Product Managers：快速验证假设
- Marketers：了解真实消费者声音

## 场景库（按人群）
- Consultants：跨文化市场洞察、品牌定位研究
- Product Managers：产品概念验证、功能优先级
- Marketers：营销信息测试、用户画像

## 案例库
- 气泡咖啡新品验证（适合PM、Marketers）
- 品牌重新定位研究（适合Consultants）
- 用户流失原因分析（适合PM）
```

**维护原则**：
- 核心模块更新 → 所有版本同步
- 可变模块独立维护
- 定期检查一致性

### Q6：演讲稿和落地页内容有什么区别？

**核心区别**：

| 维度 | 落地页 | 演讲稿 |
|------|--------|--------|
| **阅读方式** | 扫描（Scan） | 聆听（Listen） |
| **停留时间** | 3秒决定去留 | 30分钟持续吸引 |
| **信息密度** | 极高（表格、数据） | 适中（故事、类比） |
| **交互方式** | 点击、滚动 | 提问、演示 |
| **记忆点** | 视觉冲击 | 金句、故事 |

**内容设计差异**：

**落地页**：
- 用表格，不用长段落
- 用数据，不用故事（数据即故事）
- 用图标，不用文字
- 3秒内抓住注意力

**演讲稿**：
- 用故事，数据辅助
- 用类比，便于理解
- 用悬念，保持兴趣
- 30分钟持续吸引

**示例对比**：

同一个信息，不同呈现：

**落地页版本**：
```

50-100元 vs 5-10万元
[大号数字 + 对比箭头]
```

**演讲稿版本**：
```
"传统调研需要多久？
招募用户1周、执行访谈1周、分析数据1周
总计3-4周，5-10万元

atypica呢？

花费还不到300元"
