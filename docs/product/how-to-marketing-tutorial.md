## 第一部分：从代码到功能理解

### Step 1: 找到核心实现文件

**目标**：定位 Interview 和 Discussion 的实现代码

**操作步骤**：

1. **找到工具定义文件**
   ```bash
   # 打开代码库根目录
   cd /path/to/atypica-llm-app

   # 搜索 interviewChat 和 discussionChat
   grep -r "interviewChat" src/app/(study)/tools/
   grep -r "discussionChat" src/app/(study)/tools/
   ```

2. **定位关键文件**
   - `src/app/(study)/tools/interviewChatTool.ts` - Interview 工具实现
   - `src/app/(study)/tools/discussionChatTool.ts` - Discussion 工具实现
   - `src/app/(study)/agents/configs/studyAgentConfig.ts` - 两个工具都在这里被使用

3. **阅读工具描述**

   打开 `interviewChatTool.ts`，找到 description 字段：
   ```typescript
   description: `Interview one or multiple personas to understand their ...
   This tool will NOT return the interview results...`
   ```

   **关键发现 #1**：Interview **不返回结果**

   打开 `discussionChatTool.ts`，找到 description 字段：
   ```typescript
   description: `Start a group discussion with 3-8 AI personas...
   Returns a summary of the discussion...`
   ```

   **关键发现 #2**：Discussion **返回总结**

### Step 2: 提取技术参数

**目标**：找到可量化的差异点

**操作步骤**：

1. **人数限制**

   在 `interviewChatTool.ts` 中：
   ```typescript
   if (personaIds.length > 5) {
     throw new ToolCallError(
       "You can only interview up to 5 personas at a time"
     );
   }
   ```

   **数据点 #1**：Interview 每批最多 5 人

   在 `discussionChatTool.ts` 中：
   ```typescript
   personaIds: z.array(z.number())
     .min(3)
     .max(8)
     .describe("ID list of 3-8 personas to participate"),
   ```

   **数据点 #2**：Discussion 需要 3-8 人

2. **交互模式**

   在 `interviewChatTool.ts` 中找到执行逻辑：
   ```typescript
   // 串行访谈，一个一个来
   for (const personaId of personaIds) {
     await conductInterview(personaId, questions);
   }
   ```

   **数据点 #3**：Interview 是串行（一对一）

   在 `discussionChatTool.ts` 中：
   ```typescript
   // 并行讨论，所有人同时参与
   const discussion = await startGroupDiscussion(personaIds, topic);
   ```

   **数据点 #4**：Discussion 是并行（群体）

3. **输出形式**

   回顾 description：
   - Interview: "This tool will NOT return the interview results"
   - Discussion: "Returns a summary of the discussion"

   **数据点 #5**：输出可见性完全不同

### Step 3: 理解用户场景

**目标**：从技术差异推导出适用场景

**思考框架**：

| 技术特性 | 推导 | 适用场景 |
|---------|------|---------|
| Interview 不返回结果 | → AI 需要完整信息，用户不需要 | 深度挖掘个体动机，用于最终报告 |
| Discussion 返回总结 | → 需要快速了解群体共识 | 观察意见分布，快速判断趋势 |
| Interview 串行 5 人 | → 深度访谈，追问式对话 | 完整决策旅程追踪 |
| Discussion 并行 3-8 人 | → 观察群体互动和碰撞 | 多样性意见收集，权衡过程 |

**阅读真实案例**：

打开 `/docs/user-research-journey.md`，搜索 "interview" 和 "discussion"：

#### 5.6 常见问题

```markdown
## 常见问题

### Q1: Interview 不返回结果，那我怎么知道访谈了什么？
**A**: Interview 的设计逻辑是"AI 看，用户不看"。访谈内容会：
- 存储在消息流中，AI 可以继续使用
- 直接用于报告生成，用户看最终洞察
- 如果你需要看访谈过程，应该用真人访谈或其他工具

### Q2: Discussion 只返回总结，我能看到完整对话吗？
**A**: 目前不能。Discussion 只返回总结（主要观点 + 意见分布）。完整对话存储在系统中，但不对用户展示。

### Q3: 为什么 Interview 要分批，Discussion 不用？
**A**:
- Interview 是深度访谈，每个人需要完整对话，串行执行时间长，分批可以避免超时
- Discussion 是群体讨论，并行执行，时间可控，不需要分批

### Q4: 我可以同时用 Interview 和 Discussion 吗？
**A**: 可以，而且推荐。典型流程：
1. 先用 Discussion 快速判断群体态度
2. 再用 Interview 深挖有代表性的个体
3. 两者结合，既有广度又有深度

### Q5: 10 个人设，Interview 怎么分批？
**A**: 自动分成 2 批（5+5）。你只需要指定 10 个人设 ID，工具会自动处理分批逻辑。

### Q6: Discussion 最少 3 人，为什么？
**A**: 少于 3 人很难形成"讨论"，更像是对话。Discussion 的价值在于观察多样性意见和互动，至少需要 3 个不同观点。
```

**撰写要点**：
- 回答用户真实的困惑
- 基于技术实现给出准确答案
- 提供使用建议

   - 对方做得更好（客观承认）
   - 使用建议（什么时候用哪个 + 组合使用）

3. **落地页内容模板**
   - 震撼开场（3 秒抓住注意力）
   - 核心价值（数据说话）
   - 功能展示（场景化）
   - CTA（行动号召）

4. **KOL 演讲稿模板**
   - 悬念开场（提问）
   - 行业背景（痛点 + 故事）
   - 方案引入（技术 + 数据）
   - 实际演示（完整案例）
   - 能力边界（能做 vs 不能做）
   - 金句结尾（呼应开场）

**这些模板可以直接用于其他功能和竞品的文档产出。**

## 第六部分：举一反三

### 如何将本教程应用到其他功能？

**步骤**：

1. **选择功能**：例如 "Scout Agent"
2. **找代码**：`src/app/(newStudy)/` 和 `src/app/(study)/tools/scoutTaskChatTool.ts`
3. **提取参数**：3 阶段（观察 → 推理 → 验证）、社媒人设、79 分一致性
4. **推导场景**：适合快速理解用户群体，不适合深度个体研究
5. **找竞品**：对标 Brandwatch（社媒监测）、Listen Labs（AI 访谈）
6. **套用模板**：功能对比文档 + 竞品对比文档 + 落地页 + 演讲稿

**关键**：所有数据都从代码和文档中来，不要编造。

### 如何将本教程应用到其他竞品？

**步骤**：

1. **选择竞品**：例如 "NotebookLM"
2. **了解竞品**：访问官网、试用产品、阅读文档
3. **确定对比维度**：功能范围、适用场景、成本、速度
4. **找差异点**：我们有人设模拟，NotebookLM 没有
5. **客观承认优势**：NotebookLM 在文档分析和播客生成上很强
6. **套用模板**：竞品对比文档

**关键**：客观承认对方优势，说明适用场景，不贬低竞品。
